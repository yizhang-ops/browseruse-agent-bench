# v1.0.0 Benchmark Snapshot Release Checklist

Use this checklist for LexBench-Browser v1.0.0 and future benchmark snapshot releases.

## Dataset Manifest

- Confirm `browseruse_bench/data/LexBench-Browser/data_info.json` has the release version,
  default split, split files, task count, label schema, and last updated date.
- Confirm JSONL files are present:
  - `task.jsonl`
  - `task_global.jsonl`
  - `task_lexmount.jsonl`
- Check that each task has stable `id`, `query`, labels, `reference_answer`, `score_threshold`,
  and `robustness_tags`.
- Record task counts for each split in release notes.

## Validation

- Run JSON parse checks for data and metadata files.
- Run the result submission validator:

```bash
uv run python scripts/validate_result_submissions.py
```

- Run focused tests for any release tooling changed in the release PR.
- For agent-touching changes, run a real smoke:

```bash
uv run bubench run --agent <agent> --data LexBench-Browser --mode single
```

## Reference Results

- Record any maintained baseline results in `community/leaderboard/accepted-results.json`.
- Include benchmark version, split, agent version, model ID, browser backend, judge model,
  judge strategy, and rerun status.
- Link task-level artifacts or state why artifacts are not public.
- Do not silently drop failed, skipped, errored, or not-evaluated tasks.

## Changelog

- Update `browseruse_bench/data/LexBench-Browser/VERSION_HISTORY.md`.
- Summarize task additions, removals, label changes, evaluation prompt changes, and judge changes.
- Note compatibility impact for existing leaderboard entries.
- Mention known flaky sites, blocked regions, or provider incidents if relevant.

## Citation

- Confirm `CITATION.cff` points to the correct repository and release version.
- Include citation instructions in README and docs.
- If a paper/report exists later, link it from the release notes.

## License and Dataset Card

- Confirm code license is Apache-2.0.
- Confirm benchmark metadata/task definitions are declared as CC-BY 4.0.
- Confirm third-party site content, screenshots, and traces are not relicensed.
- Update the Hugging Face dataset card draft before publishing or syncing.

## Artifacts

- Attach or link:
  - dataset archive or commit SHA
  - docs URL
  - result metadata
  - release notes
  - dataset card
  - known limitations
- Avoid uploading secrets, cookies, private account state, or raw provider logs containing tokens.

## GitHub Release

- Create a release tag, for example `v1.0.0`.
- Include the benchmark version, data commit, task counts, evaluation protocol summary, and links.
- Link `EVALUATION_PROTOCOL.md`, `GOVERNANCE.md`, and `CONTRIBUTORS.md`.
- Mention how to submit results through GitHub PRs.

## Post-Release

- Verify docs deploy successfully.
- Verify package/container workflows are green or document known warnings.
- Update open issues and milestones.
- Invite official result submissions from supported agent projects.
- Monitor reported task breakages and label them against the release snapshot.
